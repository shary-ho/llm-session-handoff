# Migration Package 模板

复制以下结构，逐节填写。找不到依据的内容一律写"未知"或"当前无法获取"，不要编造。

```markdown
# [项目名称] — Migration Package

> 生成时间：[日期]
> 用途：供下一个 AI / 下一个对话窗口在不阅读历史记录的情况下直接接手本项目。

## 1. Project Overview

**项目名称：** ...

**一句话介绍：** ...

**项目目标：** 最终希望完成什么，交付形态是什么（一个 App？一份报告？一个内部工具？）。

**当前阶段：** 需求分析 / 原型设计 / Coding / Debug / 测试 / 文档 / 发布（选一个或说明跨越多个阶段）

---

## 2. Current Status

**已经完成：**
- ...

**仍未完成：**
- ...

**当前 Blocker：**
- 具体卡在什么地方，如果没有明确 blocker 就写"暂无"

**最近一次工作的内容：**
- ...

---

## 3. Directory Structure

如果可以访问项目目录，给出真实目录树（不要凭空猜测层级），并说明每个一级目录的用途，例如：

```
project/
├── README.md          # 项目说明
├── frontend/          # ...
├── backend/           # ...
├── docs/              # ...
├── config/            # ...
└── scripts/           # ...
```

如果无法访问目录，写：**当前无法自动获取项目目录，需要用户手动提供或重新上传。**

---

## 4. Important Files

| 文件 | 用途 | 是否必须阅读 |
|---|---|---|
| README.md | ... | 是 |
| ... | ... | ... |

只列真正重要、影响接手判断的文件，不要把整个目录树都塞进来。

---

## 5. Core Decisions

这是交接包里价值最高的部分之一，每一项决策都用统一格式，方便下一个 AI 直接理解"为什么是这样"，避免推翻已经权衡过的选择：

**Decision:** 采用了什么方案
**Reason:** 为什么选它
**Alternative:** 曾经考虑过的替代方案
**Conclusion:** 目前的结论（是否仍然有效，或者已经因为什么原因被取代）

（对每一项重要决策重复这个结构。只记录真正做过权衡的决策，不要把每一个小选择都列进来。）

---

## 6. User Preferences

总结用户在整个项目过程中表现出的**长期性、可复用**的偏好，例如：
- 代码风格（缩进、命名习惯、是否偏好函数式/面向对象等）
- 回复风格（喜欢详细解释还是直给结论、喜欢先方案后编码还是直接动手）
- 架构偏好（例如偏好简单直接而非过度设计）

不要记录与项目推进无关的私人信息（身份、联系方式等）。

---

## 7. Failed Attempts

**全文最重要的一节。** 这里记录的每一条都能替下一个 AI 省下重新踩坑的时间和用户重新解释一遍的耐心。

**尝试过的方案：** ...
**失败原因：** 具体是什么导致失败（技术限制？和现有架构冲突？性能不达标？用户不满意？）
**以后不要再尝试：** 明确写出下一个 AI 应该避免建议的方向

（每个失败方案单独一条，即使只有一两条也要认真写，不要因为"看起来不重要"而省略——很多重复劳动就是因为这一节被写得太笼统。）

---

## 8. Common Mistakes Made By AI

总结之前的 AI（可能是你自己在更早的对话里）在这个项目上犯过的**具体**错误，例如：
- 误解了某个需求，导致做错方向
- 重复建议了已经被否决的方案
- 修改了不相关的代码
- 忽略了已有实现，重新造轮子
- 给出了过时的方案（比如用了已经弃用的 API/库版本）

尽量具体到"在什么情况下犯了什么错"，而不是泛泛的"AI 有时会误解需求"。

---

## 9. Constraints

项目的硬性限制，例如：
- 语言/运行时版本（如 Python 3.11，不能升级）
- 平台兼容性要求（如必须兼容 Windows）
- 许可证限制（如只能使用 MIT License 的依赖）
- 部署环境限制（如只能部署内网，不能联外网）

---

## 10. Tech Stack

- **语言：** ...
- **框架：** ...
- **数据库：** ...
- **SDK / API：** ...
- **模型（如涉及 AI）：** ...
- **工具链：** ...
- **部署方式：** ...

---

## 11. APIs / Interfaces

如果项目涉及 REST / GraphQL / MCP / Tool Calling / Function Calling / SDK / 插件等接口，统一整理调用约定、关键端点、鉴权方式等，具体到可以直接拿来写代码的程度。如果不涉及，写"本项目不涉及外部接口"。

---

## 12. Open TODOs

按优先级排序：

**P0（阻塞性，必须先做）：**
- ...

**P1（重要，但不阻塞）：**
- ...

**P2（可以之后再做）：**
- ...

---

## 13. Suggested Next Prompt

**这一节会被提取为最终交付物中"用户直接复制给下一个模型的提示"，所以必须认真写。**

写一段可以直接复制粘贴给下一个 AI 的开场白，语气是对 AI 说话（"你将接手这个项目……"），而不是对用户说话。示例结构：

```
你将接手项目「[项目名]」。请先完整阅读本交接文档，再查看交付包中的项目文件，然后开始工作。

## 项目简介
[2-3 句话说明项目是什么、当前做到哪了]

## 核心规则（必须遵守）
1. 不要重新设计已确定的架构决策（见本文档 §5 Core Decisions）
2. 不要重复已经证明失败的方案（见 §7 Failed Attempts）
3. 避免之前 AI 犯过的错误（见 §8 Common Mistakes Made By AI）
4. 遵守项目的硬性约束（见 §9 Constraints）

## 当前优先级
请从 Open TODO（§12）中优先级最高的任务开始，当前最紧急的是：[具体任务]

## 协作规范
- 编码风格和用户偏好见 §6 User Preferences
- 如果某个信息本文档没有覆盖，先向用户确认，不要自行假设
- 修改代码前先确认文件在 §3 Directory Structure 中的位置和用途

现在请从 [具体的第一步行动] 开始。
```

要求：
- 整体控制在 300-500 字
- 所有 `[ ]` 占位符必须替换为实际内容
- 下一个 AI 看到这段话后，能立刻知道：项目是什么、文档在哪、规则是什么、第一步做什么

---

## 14. Attachments

列出涉及的图片、PRD、设计稿、PDF、Excel、数据库文件、Prompt 文件、配置文件、代码文件等，说明各自的作用，并标注它在打包出去的 `files/` 目录里的相对路径（例如 `files/docs/PRD.pdf`），做到文档和实际文件一一对应。如果某个文件确实无法获取（用户从未上传、或已经丢失），写明"当前无法自动获取，需要用户重新提供"——不要用摘要或复述内容冒充原文件放进 `files/` 目录。

| 文件 | 用途 | 在交付包中的路径 |
|---|---|---|
| PRD.pdf | 需求文档 | files/docs/PRD.pdf |
| ... | ... | ... |

---

## 15. Risks

- 依赖版本风险
- 性能风险
- 兼容性风险
- 尚未验证/尚未测试的模块
- 未来容易踩坑的地方（如果能预判）

---

## 16. Handoff Checklist

逐项确认后打勾，任何一项做不到就诚实地标注为未完成，不要为了好看全部打勾：

- [ ] 已说明项目目标与当前阶段
- [ ] 已说明目录结构（或注明无法获取）
- [ ] 已列出重要文件
- [ ] 已整理核心决策
- [ ] 已整理失败方案
- [ ] 已整理 AI 犯过的错误
- [ ] 已列出限制条件与技术栈
- [ ] 已按优先级列出 Open TODO
- [ ] 已给出可直接使用的 Next Prompt
- [ ] 本交接包可直接用于迁移，无需回看历史对话
```
